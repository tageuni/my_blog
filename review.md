# 2048 게임 코드 리뷰 + 플레이 테스트 결과

## 요약

`spec.md`의 설계와 실제 구현(`src/games/2048/index.html`, `style.css`, `game.js`, `build/config.js`, `build/build.js`)을 비교 검토하고, `npm run build`로 빌드한 `dist/`를 `http-server`로 서빙해 실제 브라우저(Chrome 기반 프리뷰)에서 플레이 테스트했다.

**총평**: 게임의 핵심 로직(이동/병합/3연속 병합 방지/점수/최고점수 저장/새 게임/게임오버/승리 판정/다크모드/반응형/빌드 통합)은 설계대로 정확하게 동작한다. 코드 리뷰와 콘솔 로그 확인, 수천 회의 자동화된 이동 스트레스 테스트에서 런타임 에러는 전혀 발생하지 않았다. 다만 두 가지 이슈를 발견했다 — 하나는 실제로 재현에 성공한 시각적 버그(타일 이동 애니메이션 미작동)이고, 다른 하나는 코드 추적으로 확인했으나 조건이 희귀해 GUI에서 재현하지 못한 잠재적 버그(승리+게임오버 동시 발생 시 멈춤)이다. 둘 다 게임의 핵심 플레이(이동/병합/점수)에는 영향을 주지 않는다.

## 발견된 버그/이슈

### 1. 타일 이동 슬라이드 애니메이션이 실제로 전혀 작동하지 않음 (실제 재현 확인됨)

- **파일:위치**: `src/games/2048/game.js` `render()` 함수(약 210~240줄, 특히 215줄 `tileLayerEl.innerHTML = '';`와 224~225줄 `tile.style.top`/`tile.style.left` 설정), `src/games/2048/style.css` 191줄(`.tile { ... transition: top 120ms ease, left 120ms ease; ... }`)
- **원인**: `render()`가 매 이동마다 `#tile-layer`의 `innerHTML`을 통째로 비우고 모든 타일을 새 DOM 엘리먼트로 재생성한다. 새로 생성된 엘리먼트는 `top`/`left`가 최종 위치 값으로 이미 설정된 채 DOM에 삽입되므로, 브라우저 입장에서는 "이전 값 → 새 값"으로 바뀌는 것이 아니라 처음부터 최종 위치를 가진 엘리먼트가 나타나는 것이다. CSS `transition`은 같은 엘리먼트에서 스타일 값이 변경될 때만 트리거되므로, 이 구조에서는 절대 발동하지 않는다.
- **재현 방법**: 브라우저에서 `#tile-layer`에 `transitionrun`/`transitionstart` 리스너를 걸고 방향키를 여러 번 눌러도 이벤트가 한 번도 발생하지 않음을 확인했다. 추가로 이동 전 특정 타일 DOM 노드 레퍼런스를 저장해두고 이동 후 `document.body.contains(그노드)`를 확인한 결과 `false` — 즉 이동 시 타일이 이동하는 게 아니라 매번 파괴되고 새로 생성됨을 직접 확인했다. 실제 화면에서도 타일이 스르륵 미끄러지지 않고 즉시 순간이동한다(새로 생성/병합된 타일의 `tile-pop` 등장 애니메이션만 보임).
- **영향/심각도**: **Low**. 게임 플레이(이동/병합/점수/승패)에는 전혀 지장이 없고 순수히 시각적 완성도 문제다. 다만 `spec.md` 4.7절과 `style.css`가 명시적으로 의도한 "이동 애니메이션"이 죽은 코드(`transition` 선언은 있으나 절대 실행되지 않음)로 남아있다는 점에서 설계와 구현의 괴리다.

### 2. 승리와 게임오버가 같은 이동에서 동시에 발생하면 "계속하기" 이후 게임이 응답 없이 멈춤 (코드 추적으로 확인, GUI 재현은 실패)

- **파일:위치**: `src/games/2048/game.js` `handleMove()` 함수(259~285줄), 특히 260줄 `if (isOver) return;`와 278~284줄의 조기 반환 로직
- **원인**: 한 번의 이동에서 (a) 2048 타일이 새로 만들어져 `hasWon`이 처음으로 `true`가 되고, 동시에 (b) 그 이동으로 보드가 가득 차면서 더 이상 합칠 수 있는 인접 쌍이 없어 `checkGameOver()`가 `isOver = true`로 설정하는 경우를 생각해보자. `handleMove`는 `checkWin()` → `checkGameOver()` → `render()` 순으로 실행한 뒤, `hasWon && !wasWonBefore` 조건이 참이면 승리 오버레이를 띄우고 **즉시 `return`** 한다(279~281줄). 이때 `isOver`는 이미 내부적으로 `true`가 되어 있지만, `if (isOver) { showOverlay('게임 오버', false); }` 분기(282~284줄)는 실행되지 않는다. 사용자가 승리 오버레이의 "계속하기"를 누르면 `keepPlayingAfterWin = true`로 설정되고 오버레이가 사라지지만, 그 다음 방향키 입력마다 `handleMove` 최상단의 `if (isOver) return;`(260줄)에 걸려 아무 것도 하지 않고 즉시 반환된다. 결과적으로 오버레이도 없고 에러도 없이 게임이 조용히 멈춘 상태가 되어, 사용자는 왜 반응이 없는지 알 수 없다.
- **재현 시도**: 이 상황은 "2048을 만드는 바로 그 이동이 동시에 보드의 마지막 빈 칸을 채우고 더 이상 어떤 병합도 불가능해지는" 매우 드문 케이스다. `Math.random`을 결정론적으로 고정하거나(항상 첫 빈 칸에 2 스폰) 여러 휴리스틱 우선순위 전략(좌→상→우→하 등)으로 최대 6~8천 회의 자동 이동을 여러 차례 반복 실행해 최대 512 타일까지는 도달했지만, 실제로 2048 타일 생성과 보드 만석이 겹치는 상황은 재현하지 못했다. 다만 코드 경로는 명확히 위와 같이 결함이 있음을 정적 분석으로 확인했다.
- **영향/심각도**: **Low~Medium**. 발생 확률 자체는 매우 낮지만(정상적인 플레이에서는 거의 마주치기 어려움), 실제로 발생하면 사용자에게 아무 설명 없이 게임이 완전히 멈추는 나쁜 사용자 경험이다. "승리 후 계속하기"라는 정상적인 사용자 플로우 자체는 흔하므로, 이 경계 조건만 예외적으로 방치되어 있는 셈이다.

이 외의 이동/병합 로직, 3연속 병합 방지, `moved` 플래그에 따른 신규 타일 생성 조건, 오버레이 표시/숨김의 기본 흐름, localStorage 저장/복원, 빌드 스크립트(`config.js`/`build.js`)의 정적 자산 복사 및 basePath 처리에서는 버그를 발견하지 못했다.

## 개선 제안 (필수 아님)

1. **이동 애니메이션 버그(이슈 1) 수정 방향**: `render()`에서 매번 `innerHTML`을 비우지 말고, 보드 좌표별로 타일 DOM 엘리먼트를 재사용(또는 값/좌표를 키로 한 식별)하여 기존 엘리먼트의 `style.top`/`left`만 갱신하면 `transition`이 의도대로 발동한다. 다만 이는 "매 이동마다 새로 그린다"는 현재의 단순한 렌더링 모델보다 구현이 복잡해지므로, 우선순위와 트레이드오프를 고려해 결정하면 된다.
2. **동시 승리/게임오버 버그(이슈 2) 수정 방향**: `handleMove`에서 승리 오버레이를 띄우고 조기 `return`하기 전에 `isOver`가 이미 `true`인지 함께 확인해, 승리와 동시에 게임오버가 발생했다면 "계속하기" 클릭 시점에 바로 게임오버 상태임을 안내하거나(예: "계속하기"를 승리 오버레이에서 아예 숨김), 최소한 `keepPlayingAfterWin`이 설정된 후 첫 입력에서 `isOver`이면 게임오버 오버레이를 다시 띄우도록 분기를 추가하는 방법을 고려할 수 있다.

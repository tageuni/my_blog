# CLAUDE.md

이 파일은 이 저장소에서 작업하는 Claude Code에게 프로젝트 컨텍스트를 제공합니다.

## 프로젝트 개요

마크다운(.md) 파일을 읽어 정적 블로그 웹사이트로 변환하는 프로젝트다. 별도의 백엔드 서버 없이 브라우저에서 동작하거나, 빌드 시점에 정적 HTML로 변환하는 방식을 지향한다.

## 기술 스택 / 제약사항

- **프레임워크 사용 금지**: React, Vue, Svelte 등 프론트엔드 프레임워크를 쓰지 않는다.
- **빌드 도구 최소화**: Webpack, Vite 등 번들러 없이 순수 HTML, CSS, JavaScript로 구현한다.
- 외부 라이브러리가 꼭 필요한 경우(예: 마크다운 파서)만 최소한으로 도입하고, 가능하면 직접 구현을 우선 검토한다.
- 브라우저 표준 API(ES 모듈, Fetch API 등)를 적극 활용한다.

## 디자인 원칙

- **깔끔하고 읽기 좋은 타이포그래피**: 본문 가독성을 최우선으로 한다 (적절한 줄 간격, 본문 폭 제한, 읽기 편한 폰트 크기).
- **다크 모드 지원 필수**: `prefers-color-scheme` 미디어 쿼리를 기본으로 하고, 사용자가 수동으로 토글할 수 있는 옵션도 고려한다.
- **모바일 반응형**: 모든 화면 크기에서 레이아웃이 깨지지 않아야 한다. 모바일 우선(mobile-first)으로 CSS를 작성한다.
- 불필요한 장식보다 콘텐츠 가독성을 우선한다.

## 아키텍처

마크다운 → HTML 변환은 **빌드 타임**에 이루어진다 (Node.js 스크립트, `npm run build`). 생성된 사이트 자체는 순수 HTML/CSS/JS만 사용하며, Node.js는 빌드 시점 도구로만 쓰이고 브라우저에는 노출되지 않는다.

```
posts/                # 마크다운 원본 (글 작성 위치)
src/
  styles/main.css      # 다크모드/반응형 스타일 (다크모드: CSS 커스텀 프로퍼티 + prefers-color-scheme + data-theme 오버라이드)
  scripts/
    theme-toggle.js    # 다크모드 수동 토글, localStorage 저장
    tag-filter.js      # 인덱스 페이지 태그 필터링 (클라이언트 사이드)
    search.js          # search-index.json 기반 검색 (fetch, 디바운스)
build/
  build.js             # 빌드 진입점: posts 탐색 → 파싱 → 페이지/태그/검색인덱스 생성 → 정적자산 복사
  frontmatter.js        # `---` 구분 프론트매터 파서 (직접 구현, 의존성 없음)
  markdown-parser.js    # 마크다운 → HTML 변환기 (직접 구현, 2단계: 블록 파싱 → 인라인 파싱)
  templates.js           # HTML 템플릿 함수 (문자열 리터럴 기반, 템플릿 엔진 없음)
  config.js              # 사이트 제목/네비/basePath 등 전역 설정
dist/                  # 빌드 결과물 (gitignore 대상, npm run build로 재생성)
```

**빌드/실행 방법**: `npm run build` (= `node build/build.js`) → `dist/`에 정적 사이트 생성. 로컬 확인 시 `search.js`가 `fetch()`로 `search-index.json`을 불러오므로 **반드시 HTTP 서버로 서빙**해야 한다 (`file://`로 열면 fetch가 실패한다).

**마크다운 문법 지원 범위**: 헤딩, 문단, 굵게/기울임, 링크, 이미지, 순서/비순서 리스트(1단계), 펜스 코드블록(언어 힌트), 인라인 코드, 블록인용, 수평선. 중첩 리스트/표/각주/HTML 통과는 의도적으로 미지원.

**글 프론트매터**: `title`, `date`(YYYY-MM-DD) 필수. `tags`, `description`, `slug`는 선택 (생략 시 파일명/본문에서 유추).

## 작업 시 참고사항

- 코드 작성 시 주석은 최소화하고, 자명하지 않은 이유(WHY)가 있을 때만 남긴다.
- 새 기능을 추가할 때 프레임워크나 빌드 도구 도입이 필요해 보이면, 위 제약사항과 충돌하므로 먼저 사용자에게 확인한다.

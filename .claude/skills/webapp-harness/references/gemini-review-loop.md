# Gemini(Antigravity CLI) 2차 리뷰 루프

Claude 서브에이전트의 Review 단계와 별개로, 다른 모델의 독립적인 시선을 하나 더 받고 싶을 때 쓰는 절차다. 같은 모델이 리뷰까지 하면 자기가 짠 코드의 맹점을 그대로 놓치기 쉬운데, 다른 모델은 다른 종류의 문제(예: 이 프로젝트에서는 XSS, 접근성, SEO 메타데이터 누락)를 잡아냈다 — 그래서 "선택"이지만 권장한다.

## 0. CLI를 찾는다

Antigravity(Google) CLI의 실제 실행 파일 이름은 `agy.exe`(Windows 기준)이고, 흔히 이런 경로에 있다:
- `C:\Users\<user>\AppData\Local\agy\bin\agy.exe`
- 설정/대화 기록은 `~/.gemini/antigravity-cli/`에 저장된다 (그 안의 `bin/agentapi.bat`를 열어보면 실제 exe 경로가 적혀 있어서 위치를 못 찾을 때 단서가 된다)

PATH에 없을 수 있으니, `where`/`Get-Command`로 안 잡히면 위 경로들을 직접 확인한다. 설치 자체가 안 되어 있으면(폴더가 비어있거나 없으면) 사용자에게 알리고 스킵한다 — 억지로 우회하지 않는다.

## 1. 사용 가능한 모델을 확인한다

```
agy.exe models
```
"Gemini"가 들어간 모델 중 리뷰 품질이 좋은 걸 고른다 (예: `Gemini 3.1 Pro (High)`). 모델명은 버전이 올라가면서 바뀔 수 있으니 매번 `models` 출력으로 실제 존재하는 이름을 확인하고 쓴다 — 예전에 썼던 이름을 그대로 가정하지 않는다.

## 2. 헤드리스로 실행한다

```
agy.exe -p "<프롬프트>" --model "<위에서 확인한 모델명>" --dangerously-skip-permissions
```

- `-p`(`--print`)가 비대화형 단발 실행 모드다.
- **`--dangerously-skip-permissions`는 반드시 필요하다** — 헤드리스로 서브에이전트 안에서 실행하면 대화형 권한 프롬프트에 응답할 사용자가 없어서, 이 플래그 없이는 파일을 쓰려는 순간 멈춰버린다.
- 이 플래그는 위험한 것으로 분류되어 자동 승인되지 않을 수 있다. **매번 사용자에게 명시적으로 1회 승인을 받는다** — "이번 1회에 한해 허용" 같은 답을 받은 뒤에만 서브에이전트에 이 플래그 사용을 지시한다. 예전 세션에서 승인받았어도 새 요청에는 다시 물어본다.

## 3. 프롬프트 설계

핵심 지시 3가지를 프롬프트에 반드시 포함시킨다:
1. **기존 리뷰 문서를 먼저 읽고 참고하게 한다** (`review.md`, `review-*.md` 등) — 이미 나온 지적을 반복하지 않고 새 관점(보안, 성능, 접근성, 아키텍처 등)에 집중하게 하기 위해서다.
2. **코드를 직접 수정하지 말고 리뷰 결과만 별도 파일에 쓰게 한다** (예: `gemini-review.md`) — 리뷰와 수정을 분리해서, 내(오케스트레이터)가 타당성을 검증한 뒤에 고르게 반영하기 위해서다.
3. **출력 형식**을 명시한다 (예: `# 요약 / ## 발견된 문제 (파일:라인, 근거, 심각도) / ## 개선 제안`) — 나중에 검증하기 쉽게.

## 4. 서브에이전트를 통해 실행할 때 주의할 점

- 실행 시간이 수 분 걸릴 수 있으니 타임아웃을 넉넉히 주거나 백그라운드로 돌린다.
- **서브에이전트의 결과 요약을 그대로 믿지 않는다.** 특히 결과 요약에 "instruction-shaped pattern 감지" 같은 하네스 보안 경고가 붙어 있으면, 실제로 `.claude/settings.json` 같은 민감 파일이 이번 작업 중에 바뀐 게 맞는지 직접 `git status`/파일 타임스탬프로 확인한다. (실제로는 세션 시작 전부터 있던 무관한 파일인 경우가 흔하지만, 확인 없이 넘어가면 안 된다.)
- 완료 후 반드시:
  1. `gemini-review.md`(또는 지정한 출력 파일)를 직접 `Read`해서 실제 내용을 본다.
  2. `git status --porcelain`으로 지시한 파일 외에 다른 게 바뀌지 않았는지 확인한다.

## 5. 결과를 검증하고 선별 반영한다

Gemini가 낸 지적을 곧이곧대로 다 고치지 않는다. 지적마다:
- **실제 소스 코드를 읽어서 재현 가능한지 확인한다** (예: 지적된 정규식/함수가 실제로 그 취약점을 만드는지 손으로 따라가본다).
- **고치는 데 드는 트레이드오프를 판단한다.** 명백한 버그(예: 스킴 미검증으로 인한 XSS)는 바로 고친다. 하지만 제안된 해법이 기존 기능을 약화시키거나(예: 검색 정확도를 낮추는 대가로 파일 크기를 줄이는 제안) 디자인 자산이 필요한 경우(파비콘 이미지 등)는, 조용히 바꾸지 말고 **사용자에게 트레이드오프를 설명하고 스킵하거나 확인을 구한다.**
- 수정한 뒤에는 이 하네스의 다른 원칙과 동일하게 빌드 확인 + (가능하면) 실제 브라우저로 회귀 여부를 검증한다.

## 6. 커밋

Gemini 리뷰로 나온 수정 사항은 하나의 커밋으로 묶는다. 커밋 메시지에 "Gemini 리뷰 기반으로 무엇을 고쳤고 무엇은 트레이드오프 때문에 스킵했는지"를 남겨두면, 나중에 왜 그 지적을 반영 안 했는지 다시 설명할 필요가 없다.

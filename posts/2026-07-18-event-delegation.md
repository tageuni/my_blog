# 이벤트 위임, 왜 필요할까

리스트 아이템 100개에 클릭 이벤트를 붙여야 한다면 어떻게 할까. 가장 먼저 떠오르는 방법은 반복문을 돌면서 각 아이템에 `addEventListener`를 거는 것이다. 동작은 하지만 아이템이 늘어날수록 리스너 개수도 늘고, 나중에 항목이 동적으로 추가되면 새 항목에는 리스너가 빠져 있어 클릭이 먹통이 되는 문제가 생긴다.

## 버블링을 이용하기

DOM 이벤트는 대상 요소에서 시작해 부모 방향으로 올라가며 전파된다. 이 성질을 이용하면 자식 요소 하나하나가 아니라 공통 부모 하나에만 리스너를 걸어도 된다.

```javascript
const list = document.querySelector('#post-list');

list.addEventListener('click', (event) => {
  const item = event.target.closest('li');
  if (!item) return;
  console.log('클릭된 항목:', item.dataset.id);
});
```

`event.target`은 실제로 클릭이 발생한 가장 안쪽 요소이고, `closest('li')`로 원하는 조상까지 거슬러 올라가 찾는다. 리스너는 `#post-list` 하나뿐이지만 그 안의 모든 `li` 클릭을 다 잡아낼 수 있다.

## 동적으로 추가되는 요소에도 적용된다

이 패턴의 진짜 장점은 나중에 `list.appendChild(newItem)`으로 항목을 추가해도 별도로 리스너를 붙일 필요가 없다는 점이다. 이벤트는 여전히 부모까지 버블링되므로, 새로 추가된 요소도 자동으로 같은 처리를 받는다.

이 블로그의 `tag-filter.js`처럼 태그 버튼을 눌러 목록을 필터링하는 코드를 짤 때도 같은 이유로 개별 버튼이 아니라 버튼들을 감싸는 컨테이너에 리스너를 하나만 두는 편이 관리하기 쉽다.

## 주의할 점

모든 이벤트가 버블링되는 것은 아니다. `focus`, `blur` 같은 일부 이벤트는 기본적으로 버블링하지 않으므로 위임이 통하지 않는다. 이런 경우에는 캡처링 단계를 사용하거나(`addEventListener(..., true)`), 애초에 `focusin`/`focusout`처럼 버블링하는 대체 이벤트를 쓰는 방법이 있다.

리스너 개수를 줄이고 동적 요소까지 자연스럽게 커버하고 싶다면, 이벤트 위임은 익혀둘 만한 패턴이다.
